# =========================================================
# AI DEVELOPMENT WORKFLOW
# Clean Architecture + DDD + SOLID + DRY + Atomic + Mini Functions
# =========================================================

## 0. GOAL
- Produce clean, modular, scalable code
- Enforce architecture consistency
- Keep files small, readable, testable

---

## 1. REQUIREMENT ANALYSIS

### Step 1.1 — Identify Feature
- Define feature name (e.g. "Create User")
- Define scope (what is included / excluded)

### Step 1.2 — Identify Domain
- Define:
  - Entities
  - Value Objects
  - Business Rules

OUTPUT:
- Clear domain understanding
- No coding yet

---

## 2. DOMAIN DESIGN (FIRST PRIORITY)

### Step 2.1 — Create Entity
- Define structure
- Add behavior (methods)
- No external dependency

### Step 2.2 — Create Value Objects
- Immutable
- Self-validating

### Step 2.3 — Define Repository Interface
- Only contract (no implementation)

OUTPUT:
- domain/*.entity.ts
- domain/*.value-object.ts
- domain/*.repository.ts

---

## 3. USE CASE DESIGN

### Step 3.1 — Define Use Case
- One use case = one responsibility

### Step 3.2 — Implement Use Case
- Accept input
- Validate business rules
- Call domain logic
- Call repository interface

RULES:
- No UI logic
- No direct DB/API access

OUTPUT:
- application/*.usecase.ts

---

## 4. INFRASTRUCTURE IMPLEMENTATION

### Step 4.1 — Implement Repository
- Connect to:
  - Database / API / Firebase

### Step 4.2 — Map Domain ↔ Data
- Convert entity ↔ persistence model

RULES:
- No business logic here
- Only implementation

OUTPUT:
- infrastructure/*.repository.impl.ts

---

## 5. UI IMPLEMENTATION (ATOMIC DESIGN)

### Step 5.1 — Build Components
- atoms → molecules → organisms

### Step 5.2 — Connect to Use Case
- Use hooks / handlers
- Call use case

RULES:
- No business logic in UI
- No direct API call

OUTPUT:
- ui/components/*
- ui/hooks/*

---

## 6. HELPER & UTILS EXTRACTION

### Step 6.1 — Identify Reusable Logic
- Repeated logic → extract

### Step 6.2 — Create Helper Functions
- Pure functions
- Small & reusable

OUTPUT:
- utils/helpers/*
- utils/validators/*
- utils/formatters/*

---

## 7. FILE CONTROL (MANDATORY)

### Rule Enforcement:
- Max 300 lines per file
- Ideal 100–200 lines

IF:
- File > 200 lines → refactor
- File > 300 lines → MUST split

Split into:
- smaller modules
- helper files
- sub-features

---

## 8. VALIDATION LAYER

### UI Layer
- Basic validation (required, format)

### Use Case Layer
- Business validation

RULE:
- Never trust input

---

## 9. TESTING WORKFLOW

### Priority:
1. Domain
2. Use Case
3. UI

### Step 9.1 — Test Domain
- Pure logic
- No external dependency

### Step 9.2 — Test Use Case
- Mock repository

### Step 9.3 — Test UI (optional last)

---

## 10. REVIEW CHECKLIST (MANDATORY)

Before finishing code:

[ ] Domain has no external dependency
[ ] Use case is single responsibility
[ ] No business logic in UI
[ ] Repository is interface-based
[ ] File size < 300 lines
[ ] Functions are small (<= 20 lines)
[ ] No duplicated logic
[ ] Naming is clear and consistent
[ ] Code is readable (AI + human)

---

## 11. AI GENERATION RULES

When AI generates code:

- MUST start from Domain
- MUST follow layering:
  Domain → Use Case → Infrastructure → UI
- MUST split files if too large
- MUST use interfaces
- MUST avoid mixing concerns

---

## 12. REFACTORING WORKFLOW

Trigger refactor IF:
- File > 200 lines
- Function > 20 lines
- Logic duplicated
- Hard to read

Refactor steps:
1. Extract function
2. Extract module
3. Split by responsibility
4. Improve naming

---

## 13. DELIVERY OUTPUT STRUCTURE

Feature output MUST follow:

feature-name/
  domain/
  application/
  infrastructure/
  ui/
  utils/

---

## 14. ANTI-PATTERN CHECK

DO NOT ALLOW:
- God file (>300 lines)
- Direct DB access from UI
- Business logic in UI
- Massive service class
- Tight coupling

---
